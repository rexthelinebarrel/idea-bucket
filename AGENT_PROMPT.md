# 给工作 Agent 的上下文 Prompt

> 使用方式：将以下内容粘贴给接手本项目的工作 agent（如 Claude、GPT 等），让它在开始编码前理解项目背景和设计意图。

---

## 项目起源

这个项目来自一次关于"想法太多脑子太乱"的讨论。项目创建者 rexthelinebarrel 是一个动手能力很强的创作者，日常在 Windows 平台用 Node.js / React 等技术做 side project（比如 bilibili 直播录制+高光切割工具）。他遇到的问题不是"没想法"，而是：

1. 灵感来得随机（走路、洗澡、临睡前），转瞬即逝
2. 打字记录摩擦太大，语音是最自然的捕捉方式
3. 记下来的灵感是孤立的，回头不知道从哪个开始
4. 没有 AI 搭档来 bounce ideas，想法就烂在笔记里了

## 对话历程（关键决策记录）

以下是项目创建者与产品讨论 agent 的对话摘录。每一个设计决策都有它的来由。请理解这些"为什么"，而不仅仅是"是什么"。

---

### 起点：脑子很乱，想 vibecoding 但没想法

> **创建者**：我想 vibecoding 一些便于生活的项目，但是总是感觉脑子很乱没什么想法。
>
> **agent**：脑子乱很正常，这不是缺想法，是想法没被外化。建议先建一个"痛点清单"，遇到任何"烦死了要是能自动化就好了"的瞬间立刻记下来。
>
> **创建者**：我有个想法，做一个语音输入的 APK，在移动端每天有灵感的时候记录自己 vibecoding 的想法，跟我讨论讨论。

关键洞察：创建者要的不是笔记软件，而是一个**能跟他讨论灵感的 AI 搭档**。

---

### 产品定位：随身灵感桶

> **agent**：你觉得它的第一身份是什么？随身灵感桶？创作搭档？还是个人点子日志？
>
> **创建者**：我觉得是随身灵感桶，一个项目可能需要很多点子，所以一个收集他们并且连接的容器就很重要了。

**"容器"是这个产品最核心的隐喻。** 不是文件夹，不是列表，是一个有空间的、灵感在里面互相接触的物理容器。

---

### AI 连接的三种方式

> **agent**：我脑补三种连接场景——合并（相似灵感合并）、演化（同一灵感生长轨迹）、碰撞（不同灵感化学反应）。这三种你觉得哪种最兴奋？
>
> **创建者**：我觉得这三种都需要，当 AI 检测出不同灵感可能连接的方式的时候，可以每天定时发邮件或者是通知提醒。

三种连接全部保留，且**通知是定时汇总而非实时打断**。

---

### 全局大桶 vs 分项目桶

> **agent**：一个全局大桶还是多项目桶并存？
>
> **创建者**：我认为应该是个全局大桶。

所有灵感在同一个空间里，项目只是桶上方的视角。这影响了数据结构设计和图谱的全局性。

---

### 开工场景：A+C 组合

> **agent**：你什么时候会打开这个桶？准备开工时（A）、灵感枯竭时（B）、收到通知时（C）、碎片时间（D）？
>
> **创建者**：我的感觉是 A 和 C 一起，每次灵感产生碰撞的几率较高的时候就可以有提醒，这样开工的时候就不会乱翻了。

**通知负责拉回，浏览负责高效开工。** 列表默认排序应该是 AI 推荐优先，不是时间倒序。

---

### 投入方式：极简派

> **agent**：极简派（按住说话松手即走）还是缓冲派（看一眼转写文字确认再走）？
>
> **创建者**：用极简派。

投入瞬间不可有任何摩擦。转写错误可以事后修正，但投入动作必须一气呵成。

---

### 图谱：Obsidian 风格，辅助视图

> **agent**：你觉得图谱在灵感桶里是主界面、辅助视图、还是混合？
>
> **创建者**：不不不，一点进去先是投入灵感的极简大按钮，再然后详情内是列表默认，辅助视图才是 Obsidian 那种关系图。

主界面 = 录音按钮。列表是默认浏览视图。图谱是辅助 tab。这个层次关系很重要。

---

### 列表排序：都要

> **agent**：列表默认排序按推荐优先、时间倒序、还是状态？
>
> **创建者**：列表视图的排序都要。

三种排序方式并存，但**推荐优先是默认**。

---

### 电脑端边界

> **agent**：电脑上冒灵感怎么办？
>
> **创建者**：电脑场景是需要的，但是以防工作 agent 冲突，我的这个项目可以只在电脑上呈现灵感连接，或者是帮写连接后的项目 .md。

电脑端定位为**只读+导出**。不输入灵感（防 agent 冲突），不写代码（交给工作 agent）。只做图谱浏览和生成项目文档。这个边界划分是经过深思熟虑的。

---

### 标题算法

> **agent**：标题是 AI 提炼还是当场生成？
>
> **创建者**：我们写个算法，按照关键词作为题目就好，但是要包含日期。当场就有即可。

标题在投入瞬间由本地算法生成，不等 AI。格式：`[MM/DD] 关键词1·关键词2·关键词3`。算法逻辑：分词→去停用词→词性权重排序（名词>动词>形容词）→按原文顺序排列。

---

### "200 条翻列表说明不是客户"

> **agent**：当桶里有 200 条灵感的时候，你怎么找东西？
>
> **创建者**：如果桶里有 200 条灵感我还会去看，那么说明肯定不是我们的客户。

这是定义产品成功标准的关键一句话。**桶越大，用户越不应该翻。AI 替你翻。** 列表是 AI 的推荐流，不是检索工具。

---

### 反悔期

> **agent**：灵感可以删吗？
>
> **创建者**：可以列一个反悔期。

不真正删除，有反悔窗口。灵感只是时机未到。

---

### 两端职责的最终确认

> **创建者**：手机端语音投入 + 列表浏览 + 图谱辅助。电脑端只做图谱浏览和导出项目 .md。这是第二个版本了，还是先聊移动端吧。

v1 专注移动端。v2 才上电脑端。

## 产品是什么

**灵感桶 (Idea Bucket)** — 一个移动端优先的灵感捕捉 + AI 连接工具。

一句话描述：按住说话，松手就走。AI 后台帮你发现灵感之间的关联，每日推送。开工时打开看 AI 推荐，而不是自己翻列表。

## 用户画像

- 独立创作者 / side project 爱好者
- 每天冒出多个点子，大多跟自动化、工具开发、效率提升相关
- 移动端（手机）用于灵感捕捉，电脑端用于深度工作
- 中文为主，偶尔夹杂英文技术术语
- 偏好极简交互，讨厌多余的点击和确认步骤

## 产品核心理念（请内化）

1. **极简投入 > 完美记录**：记录灵感只需 5 秒，不许任何东西挡在"说"这个动作前面。转写错误可以事后修正，但投入瞬间不可有摩擦。

2. **桶不是收纳盒，是策展人**：笔记软件让你自己翻，灵感桶让 AI 替你翻。桶越大，用户的翻找行为越少。列表是 AI 的推荐流。

3. **灵感之间会自己对话**：Obsidian 图谱是手动连的，灵感桶是 AI 自动发现连接的。用户打开图谱，看到的是被 AI 照亮的潜意识地图。

4. **手机投入，电脑开工**：手机端只管捕捉和浏览，不写代码。电脑端（v2）只做查看连接和导出项目文档，也不写代码。写代码交给专门的工作 agent——这是刻意的边界划分，防止 agent 功能冲突。

5. **灵感不会"没用"**：不真正删除灵感，只有反悔期。灵感只是时机未到。

## 设计原则

- **首屏只有一个按钮**，任何多余元素都需要论证必要性
- **投入动作不可被打断**，不需要确认步骤
- **通知要有信息量**，不推送空洞数字（"3 条新灵感"），而是推送具体发现（"你的 A 和 B 可能是同一件事"）
- **图谱是辅助视图**，默认从列表进入，不是主界面
- **列表默认排序是 AI 推荐**，不是时间倒序

## 当前状态快照（2026-07-21 v0.7.0，新会话来此时先读这段，可省大量上下文）

- 仓库：https://github.com/rexthelinebarrel/idea-bucket （master，MIT，公开）
- 技术栈：Expo SDK 57 / RN 0.86 / TypeScript，expo-router（src/ 目录），expo-sqlite + expo-file-system，expo-audio，react-native-svg（图谱）
- **转写三模式**（设置页可切）：
  - 离线引擎（默认）：react-native-sherpa-onnx。模型三选一（src/lib/offline-stt.ts）：流式 zipformer 轻量 22MB / 双语 188MB（OnlineRecognizer 分块流式解码）；**Qwen3-ASR 0.6B int8 高精度 940MB（0.7.0 起，OfflineRecognizer 整体解码，maxTotalLen 4096 / maxNewTokens 512 / numThreads 4）**。两类引擎路径不能混用（流式模型进离线识别器报维度错），按 OfflineModel.engine 字段分流。模型经 hf-mirror 下载
  - 系统识别：@jamsch/expo-speech-recognition（识别服务可手动切换）
  - 云端 API：OpenAI 兼容（硅基流动/Groq/OpenAI 均可，设置页可配）
- 已实现：极简录音入桶（**按住上滑=松手取消**，靠扩大 pressRetentionOffset 防误判，0.7.0 起）、本地标题算法（src/lib/title.ts）、列表（推荐/时间/状态排序+搜索筛选）、详情（回放/AI 讨论/手动关联/状态流转/AI 建议关联确认）、30 天回收站、SQLite 日志（环形 500 条）、诊断+一键上报、**AI 关键词提取（第三方 API，失败回退本地算法）+ Jaccard 候选 + LLM 反迎合终审 + 灵感图谱 /graph（力导向布局，实线=已连接/虚线=AI 建议）**、全端 UI 改版（发丝边框/琥珀淡底选中态/圆角体系，src/theme.ts 有 radius 常量）
- **更新机制**：仓库根目录 release.json（经 jsDelivr 拉取，多镜像取最大版本容错）→ 应用内下载 APK → 系统安装器确认；版本号唯一来源 `src/version.ts`（app.json 同步）；出包 `npx --yes eas-cli build --platform android --profile preview --non-interactive --wait`（EAS owner: nachdenken，已登录）；出包后更新 release.json 推送并调 purge.jsdelivr.net 刷新
- 版本规则：每次发版必递增（修复=补丁位 / 功能=次版本位 / 破坏=主版本位）
- **教训（勿重复踩坑）**：
  1. react-native 没有 ErrorUtils 具名导出（只在 globalThis 上，曾因它 OTA 变砖）
  2. sherpa-onnx 离线识别器不吃流式模型（报维度错误 Got: xxx Expected: 39），反之亦然——按模型 engine 分流
  3. 版本号参与 fingerprint 计算 → EAS Update OTA 与"版本号递增"结构性冲突 → OTA 已弃用
  4. eas update 必须带 `--platform android`（本项目不发 web 端）
  5. Pressable 默认 pressRetentionOffset 只有 20px，做滑动手势必须手动扩大，否则滑动途中触发 onPressOut
  6. `npx eas login` 要写 `npx eas-cli login`；expo.dev 的 APK 直链下载慢，App 内下载用 createDownloadResumable+重试+浏览器兜底
  7. Qwen3-ASR 默认 maxTotalLen 512/maxNewTokens 128 只够十几秒音频，长录音必须调大
- 待办：见 PROJECT.md「更新计划（backlog）」
